When read-only
wasn't good enough.

Designing a two-role edit and review workflow for the B2B Digital Catalogue — because AI enrichment can't fix bad data at the source.

Role: Product Designer (End-to-End) Tool: Figma Status: In Development Users: Editors & Reviewers
> BEFORE_THE_SCREENS
The best products don't fight their constraints.
They design within them.
When AI can't fix a problem,
the answer isn't a smarter algorithm.
It's giving humans the right tools.
> 01 · THE_PROBLEM

Better AI couldn't solve
a data quality problem at the source.

The B2B Digital Catalogue relied on AI enrichment to populate product attributes — manufacturer, brand, classification, country of sale. But the enrichment was only as good as the source data. Data coming in from client funnels was incomplete, inconsistent, and often wrong.

The result: catalogue records full of bad data, and no way for anyone to fix it. Product attributes were read-only. There was no edit button. No governed path to correction.

AI enrichment failed.

The model could only work with what it was given. Bad input data meant bad output — consistently.

Source data was poor.

Data arrived from multiple client funnels — incomplete, inconsistent, often missing key attributes entirely.

Clients had no way to fix it.

Attributes on the PDP were read-only. No edit button. No escalation path. No self-serve correction.

We could've trained a better model. We could've added more data sources. None of it would've worked. The problem wasn't the AI.

> 02 · WHY_AI_COULDNT_FIX_IT
The fix wasn't smarter automation.
It was structured human input.

Retraining the model, adding more data sources, tweaking thresholds — none of these approaches addressed the root cause. The problem was that no human could correct what the AI got wrong. The data entered the system incorrect and stayed incorrect.

The solution required giving clients the ability to fix attributes directly — with governance to ensure quality, and a review layer to prevent new errors from entering the system.

> 03 · USERS

One product.
Two roles. One shared outcome.

EDITOR

Makes the edits.

Client-side user who identifies incorrect attributes and submits corrections. Works inline on the PDP — no separate interface.

REVIEWER

Approves or rejects.

Client-side user who reviews submitted edits with full context. Can approve all or reject with field-level comments explaining exactly what needs fixing.

Both roles are client-side — this is a self-serve workflow. The separation of roles ensures no single user can both submit and approve their own changes, maintaining data integrity without requiring internal oversight.

Separation of roles. Shared accountability for cleaner data.

> 04 · PROCESS

The constraints weren't obstacles.
They were the design brief.

Before touching Figma, I mapped every backend constraint, every system boundary, and every edge case. The edit feature sat at the intersection of multiple products — each with their own data ownership rules.

Specifications — owned by MMA (a separate product). Cannot be edited here.

Classification path — owned by Taxonomy (another product). Read-only in this flow.

Adding new attributes — introduces data governance issues. Not in Phase 1 scope.

Partial acceptance — backend limitation. Reviewer must accept all or reject all in Phase 1.

Phase 1 ships what's safe. Phase 2 expands what's possible.

> 05 · KEY_ITERATION

The decision that changed
after client feedback.

The reviewer experience went through multiple iterations — primarily the rejection flow. Early designs allowed reviewers to accept or reject a submission as a whole, with a single comment for context.

After a client demo, the feedback was clear: "How will the editor know what specifically to change?" A single rejection comment wasn't enough. Editors were re-submitting with the same issues because they couldn't tell which fields were wrong.

BEFORE — Single rejection
  • One accept/reject on the entire submission
  • Single comment field for reviewer notes
  • Editor receives vague feedback
  • Multiple re-submission loops with same errors
AFTER — Field-level comments
  • Comment field per attribute being reviewed
  • Editor knows exactly which field to fix
  • Fewer re-submission cycles
  • Faster resolution, cleaner data outcomes

The best iterations come from real users, not internal reviews. Approval flows aren't binary — good rejections teach.

> 06 · THE_TRIGGER

From read-only to editable
in one click.

The edit flow begins on the Product Details Page — the same page where incorrect data lives. No navigation to a separate editing interface, no context switch. The edit button lives where the problem is visible.

Edit button on PDP — permission-aware, only visible to users with editor role. No separate interface to navigate to.

Same context — editor sees the incorrect data and corrects it in the same view. No spatial disorientation.

Editing lives where the data lives.

Preview before submit.

Before a submission reaches the reviewer, the editor sees a preview — original value vs edited value, side by side. One last check before the review loop begins.

Review your own work first. The reviewer reviews the result, not the typos. This preview step catches obvious errors before they enter the review queue.

> 08 · REVIEWER_FLOW

Approve, reject,
or guide.

Pending reviews queue — every submission visible with editor name, product, and timestamp. Reviewers always have full context before opening a submission.

Field-level comments on rejection.

The most important iteration in the entire flow. Reviewers can leave a comment per attribute — not a single blanket rejection. Editors receive exactly what needs fixing, field by field.

This was the most debated design decision. Early designs had a single comment field. Client feedback after demo made the problem clear — editors were re-submitting with the same errors.

Accept All

Changes go live immediately. AI enrichment re-triggers on the updated data. Audit log captures the reviewer's decision.

Send Back

Editor receives field-level feedback. Flow loops back to the edit step. No partial acceptance in Phase 1 — backend constraint.

A multi-team escalation became a 2-role self-serve workflow.

> 11 · LIMITATIONS

What we didn't
get to solve — yet.

No partial acceptance.

Reviewer must accept all or send all back. Backend limitation in Phase 1 — granular field-level acceptance is a Phase 2 problem.

No bulk editing.

Power users managing hundreds of products will need bulk correction tools. Single-product editing doesn't scale for large catalogues.

Field expansion owned elsewhere.

Adding new attribute types still requires Taxonomy team involvement. Self-serve field creation isn't in scope.

Phase 1 ships. Phase 2 learns. That's how good products grow.

> 12 · REFLECTION

Designing within constraints isn't compromising — it's collaborating with reality.

The best designers don't ask for the perfect canvas. They ask: "What's the smallest, sharpest version of this that solves a real problem today?"

This case study won't win design awards. But it'll quietly improve thousands of catalogue records — and make the AI that depends on them actually useful.

That matters more.
